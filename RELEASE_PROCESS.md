# Процесс выпуска обновлений MVS

## Обзор системы

MVS использует **electron-updater** для автоматических обновлений без участия пользователя. Система поддерживает:

- ✅ Автоматическая проверка обновлений на запуск приложения
- ✅ Проверка каждый час (фоновая)
- ✅ Красивый диалог с прогрессом загрузки
- ✅ Скачивание в фоне с показом прогресса (%)
- ✅ Автоматическая установка и перезапуск
- ✅ Поддержка пользовательского сервера (`https://updates.mvs.app/releases`)
- ✅ Логирование всех событий в `startup.log`

## Архитектура

### 1. **Главный процесс (Electron Main)**
Файл: `electron/main.ts`

```typescript
- setupAutoUpdater()    // Инициализация при запуске
- Проверка на старте:   autoUpdater.checkForUpdates()
- Проверка каждый час:  setInterval(..., 60 * 60 * 1000)
- События:
  - update-available    → отправляет renderer версии
  - download-progress   → отправляет процент (%)
  - update-downloaded   → готово к установке
  - error              → обработка ошибок
```

### 2. **Preload API (Безопасный мост)**
Файл: `electron/preload.ts`

```typescript
window.electron.updater = {
  onUpdateAvailable(callback)  // Слушать доступность обновления
  onDownloadProgress(callback) // Слушать прогресс
  onUpdateDownloaded(callback) // Слушать завершение
  onError(callback)            // Слушать ошибки
  
  checkForUpdates()   // Принудительная проверка
  downloadUpdate()    // Начать загрузку
  installUpdate()     // Установить и перезапустить
  dismissUpdate()     // Отклонить обновление
}
```

### 3. **UI Компонент (React)**
Файл: `src/components/UpdateDialog.tsx`

- Отображает текущую и новую версию
- Показывает прогресс-бар при загрузке
- Поддерживает светлую и тёмную тему
- Две фазы:
  1. **Загрузка**: "Обновить сейчас" / "Напомнить позже"
  2. **Готово**: "Перезапустить и установить" / "Позже"

### 4. **Конфигурация**
Файл: `package.json`

```json
"publish": [{
  "provider": "custom",
  "url": "https://updates.mvs.app/releases",
  "channel": "latest"
}]
```

## Процесс выпуска нового обновления

### Шаг 1: Подготовка кода

```bash
# 1. Внесите нужные изменения в код
# 2. Обновите версию в package.json
nano package.json
# Измените версию: "version": "1.0.1" → "1.0.2"

# 3. Проверьте, что всё компилируется
npm run build
```

### Шаг 2: Сборка дистрибутива

```bash
# Выполните полную сборку Electron
npm run build  # или npm run dist

# Это создаст:
# - build/win-unpacked/  (распакованное приложение)
# - MVSSetup.exe         (установщик)
# - MVSSetup.exe.blockmap
# - latest.yml           (метаинформация версии)
```

### Шаг 3: Структура артефактов на сервере

Сервер `https://updates.mvs.app/releases` должен иметь эту структуру:

```
https://updates.mvs.app/releases/
├── latest.yml                 # Метаинформация текущей версии
├── MVSSetup-1.0.1.exe        # Установщик полного обновления
├── MVSSetup-1.0.1.exe.blockmap
├── MVSSetup-1.0.0.exe        # Предыдущие версии (опционально)
└── delta/                     # Дельта-обновления (опционально)
    └── 1.0.0-1.0.1.exe
```

### Шаг 4: Содержимое `latest.yml`

Файл создаётся автоматически при сборке. Пример:

```yaml
version: 1.0.2
files:
  - url: MVSSetup-1.0.2.exe
    sha512: abc123def456... (64-символьный хеш)
    size: 125432890
path: MVSSetup-1.0.2.exe
sha512: abc123def456...
releaseDate: '2024-01-15T10:30:00.000Z'
```

### Шаг 5: Загрузка на сервер

```bash
# SSH на сервер обновлений
scp build/latest.yml user@updates.mvs.app:/releases/
scp build/MVSSetup-*.exe user@updates.mvs.app:/releases/
scp build/MVSSetup-*.exe.blockmap user@updates.mvs.app:/releases/

# Или используйте FTP/S3/другой метод деплоя
```

### Шаг 6: Проверка

1. **В приложении:**
   - Перезагрузите MVS
   - Должно автоматически проверить обновления
   - Если есть новая версия → появится диалог
   - Нажмите "Обновить сейчас" → загружается в фоне
   - Когда загрузится → "Перезапустить и установить"
   - Приложение перезагрузится с новой версией

2. **В логах:**
   - Откройте `%APPDATA%/MVS/startup.log`
   - Должны быть записи:
     ```
     [updater] Check for updates...
     [updater] Update available: 1.0.1 → 1.0.2
     [updater] Download started...
     [updater] Downloaded: 45% → 100%
     [updater] Update downloaded, ready to install
     ```

## Версионирование

- **Мажор** (1.x.x): Крупные изменения UI/функционала
- **Минор** (x.1.x): Новые функции, совместимо с предыдущими

- `1.0.0` → `1.0.1` (патч: исправление)

- `1.1.0` → `2.0.0` (мажор: полный редизайн)

## Откат версии (если нужно)

Если выпущена плохая версия:

```bash
# 1. Верните код к старой версии
git revert <commit-hash>

# 2. Измените version в package.json на текущую
# Пример: если текущая 1.0.2, верните на 1.0.1

# 3. Пересоберите
npm run build

# 4. Загрузите старую версию на сервер
# Обновите latest.yml с ссылкой на предыдущую версию

# Текущие пользователи получат откат
```

## Частые вопросы

**Q: Что если пользователь отклонит обновление?**
A: Приложение продолжит работать с текущей версией. Проверка повторится через час.

**Q: Может ли пользователь отключить автоматические обновления?**
A: Нет, это встроено. Но может отклонить конкретное обновление на 1 час.

**Q: Работает ли обновление без интернета?**
A: Нет, требуется подключение для загрузки нового exe.

**Q: Может ли пользователь откатиться?**
A: Может вручную установить предыдущую версию, но не через приложение.

**Q: Как часто проверяются обновления?**
A: На старте + каждый час в фоне.

**Q: Что если сервер обновлений недоступен?**
A: Ошибка логируется, но приложение продолжает работать. Проверка повторится позже.

## Техподдержка

При проблемах проверьте:
1. `startup.log` на компьютере пользователя
2. Доступность сервера `https://updates.mvs.app/releases`
3. Корректность `latest.yml` (проверьте SHA512 хеши)
4. Версия в `package.json`

## Развёртывание сервера (будущее)

Когда готово развёртывать на настоящий сервер:

```bash
# 1. Купите/настройте https://updates.mvs.app
# 2. Создайте директорию /releases с правами записи
# 3. Настройте HTTPS (обязательно!)
# 4. Тестируйте с локальной версией перед production
# 5. Документируйте процесс деплоя (GitHub Actions / CI)
```
