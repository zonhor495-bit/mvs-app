# Инструкции по публикации сайта и релизов (подготовка)

Этот документ описывает, какие шаги уже подготовлены в репозитории и что должен сделать владелец репозитория для окончательной публикации и проверки автообновлений.

ВАЖНО: Никогда не вставляйте персональные токены в публичные чаты. Секреты добавляются в Settings → Secrets → Actions.

Что уже подготовлено (в репозитории)
- Добавлен workflow для деплоя сайта на GitHub Pages: `.github/workflows/deploy-site.yml`.
  - На push в `main` выполняется `npm ci`, `npm run build` и деплой содержимого `dist/` в Pages.
- Добавлен workflow для сборки и публикации Electron Release: `.github/workflows/publish-electron.yml`.
  - Триггер: push тега `v*.*.*` или создание Release в GitHub.
  - Workflow запускается на `windows-latest`, собирает web-часть (`npm run build`), потом запускает `npx electron-builder build --win nsis --x64 --publish always`.
- `package.json` уже содержит конфигурацию `build.publish` для GitHub (проверьте поля `owner`/`repo`).

Что нужно настроить владельцу репозитория
1. Добавить секрет `GH_TOKEN` в `Settings → Secrets → Actions` с правами на создание релизов (scope `repo` или `public_repo` для публичного репозитория).

2. Убедиться, что в `package.json` корректно указаны поля для публикации (owner/repo/channel). Пример:

```json
"build": {
  "publish": [{
    "provider": "github",
    "owner": "<your-github-username>",
    "repo": "<repo-name>",
    "channel": "latest"
  }]
}
```

3. Для публикации сайта (GitHub Pages):
- В repository → Settings → Pages убедиться, что источник публикации разрешён (Pages автоматически создаётся через workflow). Не требуется вручную выбирать ветку, если используете `actions/deploy-pages`.

4. Для публикации релиза и автоматической генерации `latest.yml`:
- Создайте git tag, например `v1.0.2`, и запушьте его:

```bash
git tag v1.0.2
git push origin v1.0.2
```

или создайте Release через веб-интерфейс — workflow отработает при создании Release.

5. Дополнительно (рекомендуется):
- Настроить коды подписи (Windows code signing) если требуется — electron-builder тогда будет подписывать .exe; потребуется добавить секреты для сертификатов.

Проверка после действий владельца
1. После push в `main` — открыть раздел Pages (или URL `https://<your-user>.github.io/<repo>/`) и проверить, что сайт доступен.
2. После создания тега / Release — в разделе Releases должен появиться Release с артефактами: `MVSSetup.exe`, `MVSSetup.exe.blockmap` и `latest.yml`.
3. Проверьте, что `latest.yml` доступен по HTTPS по ссылке из релиза (она будет видна в списке assets).

Как проверить автообновление (кратко)
1. Установите приложение из скачанного `Setup.exe` (версия A).
2. Создайте новую версию в `package.json`, соберите и опубликуйте Release с новой версией (версия B) — через tag или Release.
3. Запустите установленную версию A — electron-updater должен обнаружить `latest.yml`, скачать инкремент и применить обновление до версии B.
4. Проверки: скриншоты логов `App`/`Updater` и сообщение о завершении обновления.

Если желаете, после добавления `GH_TOKEN` и подтверждения прав, я выполню запуск релиза / деплой и полностью проверю сценарии (скриншоты и ссылки).
